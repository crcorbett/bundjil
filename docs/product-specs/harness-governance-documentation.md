# Harness-governance documentation

## Outcome

Bundjil maintainers can retrieve the current owner for documentation,
architecture, intent, operations, proof, research, skills, and history without
loading completed rollout evidence or treating repository prose as current
provider truth. Material slices update the earliest durable owner and retain
bounded evidence, limitations, rollback, and exact Git identity.

## Invariants

- `docs/README.md` is the sole maintainer lifecycle/truth-layer router.
- Active intent requires this current SPEC/task owner and a matching active
  plan. Completed ledgers/plans remain separately reachable as history.
- Root and package READMEs contain purpose, public boundaries, and public
  commands—not undated deployment readiness, revisions, webhook topology, or
  provider actuality.
- Repeatable consequential operations live in target-owned runbooks with
  preconditions, authority, steps, evidence, rollback, and escalation.
- Proof names the artifact, environment, exact journey, observation,
  limitations, non-claims, and rollback identity.
- Effect code remains Schema-owned at boundaries, typed, flat, sequential,
  composable, and free of generic SDK callbacks, raw semantic primitives,
  `instanceof` policy, unchecked outputs, and helper sprawl.

## Non-goals and authority

This campaign does not itself change or prove Vercel, Executor, Sendblue,
Upstash, AI Gateway, deployment, webhook, credential, release, production, or
external-consumer state. Those claims require current external readback under
explicit authority and the target-owned runbook.

HGI-300-S2 removes operator procedure and dated provider/deployment proof from
app and package READMEs without creating premature runbook or verification
owners. The retained completed SPEC/plan corpus preserves rollout provenance;
it is not a substitute for the pending HGI-303 runbook, HGI-304 authority, or
HGI-305 verification owners.

HGI-301 reconciles the current Eve architecture to source-backed topology and
invariants. Its dated read-only reconciliation is addressable at
[`HGI-301-eve-reconciliation.json`](../documentation-audit/HGI-301-eve-reconciliation.json);
it records that provider readback was unavailable in this implementation
environment and therefore makes no production claim. Its semantic contradiction
fixtures are executable acceptance evidence and remain available for HGI-302's
broader future docs control.

HGI-308 has a local, acceptance-pending candidate at
[`HGI-308-source-receipt.json`](../documentation-audit/HGI-308-source-receipt.json)
and its five schema-valid decision records. It defers the Eve Production claim,
accepts app-owned runbooks and target-specific fallback readbacks, retains
occurrence-based boundary exceptions pending equivalent fixtures, and retains
Claude automatic review only as a bounded experimental signal. That
tighten-or-retire policy is the accepted default. It is not an
accepted authority, provider receipt, runbook, workflow change, or proof of
automated-review effectiveness. HGI-302 through HGI-306 implement only the
task-blocking decisions and non-claims recorded in those candidates.
